Our Philosophy

Maidaan is built around three simple beliefs: play is social, access should be simple, and pricing should be transparent. We aim to make it effortless for people to find venues, assemble players and play — without friction, hidden fees, or confusing booking flows. Our design focuses on human connection: discover local games, meet people nearby, and make sports a regular part of community life.

Design principle: small micro-interactions (hover glow, tactile buttons), accessible typography (Inter), and a neon-dark theme for energy while offering a light mode for daylight use.

Our Goal

  • Make venue discovery & booking as fast as ordering food.
  • Enable players (solo or partial teams) to find each other, post quick invites and join games instantly.
  • Provide turf owners with easy onboarding and a simple dashboard to manage availability, prices and bookings.
  • Keep pricing transparent — no hidden fees — while enabling low-friction secure payments.

Project purpose & target audience

Purpose: Provide a single place for sport players and venue owners to meet, book and play. Reduce friction in last-minute game formation and make small group sports social & recurring.

Primary users: casual players (students/professionals), solo players seeking games, amateur teams, turf owners and managers, local coaches.

Core features (detailed)

Search & Discovery

  • Search by sport, location and availability (today, morning, evening).
  • Filters: surface (grass/clay/hard), indoor/outdoor, lighting, equipment rental, coaching available, parking.

Book Now (UX)

  • Venue listing card: photos carousel, summary, price per hour, quick "Book Now" and "Find Players".
  • Click "Book Now" → opens detailed venue page in new tab containing: gallery, detailed amenities, embedded Google Maps, share button, available time slots and a booking panel (select sport if multi-use, date, start time, duration).
  • Booking panel adds selection to an on-page cart (side panel) — user confirms payment there. Cart persists during session.

Find Players

  • Post-your-game flow: "+ Post Your Game Invite" opens modal that asks: sport (dropdown incl. Squash & Pickleball), date & time, number of players required, optional venue, short message.
  • Invite appears instantly in "Game Invites" feed (no page reload) with Join & Chat actions.
  • Join button: sends a join request notification to the poster & toggles to "Request Sent".
  • Chat: lightweight chat modal per session. Real-time via WebSockets/Firebase or fallback to polling/localStorage for demo mode.

Spotify integration

Embed Spotify playlists on Booking/venue pages so players can tune-in while playing. Playlist is optional and togglable by user (autoplay muted by default).

Admin / Turf Owner

  • Turf onboarding page: form + photo uploader + request-a-demo button.
  • Owner dashboard: availability calendar, price management, booking list, payout & feedback.

Accessibility & Localization

  • Inter font, ARIA labels, keyboard-friendly focus outlines, alt-text for images.
  • Language toggle (EN / HI) — UI copy switched via a small dictionary; keep translations minimal to start and expand content later.

How it works — end-to-end flows

1) Booking flow (user)

  1. User finds venue via search or sport tile → clicks venue card → opens detailed venue page in new tab.
  2. User selects sport (if multi), date, start time, duration — availability validated in real-time.
  3. Selection added to the cart panel on the same page. User can add multiple slots or venues.
  4. Checkout → payment gateway (Stripe/Razorpay) integration — after success, booking confirmation email & in-app booking record.

2) Post & join game (players)

  1. Post: user clicks "+ Post Your Game Invite" → fills sport, time, players needed, optional venue → submit.
  2. Invite shown in feed. Other users click "Join" → sends join request to poster (in-app notification + optional email/SMS).
  3. Poster accepts or rejects requests via a small management UI (accept triggers confirmation message & updates joined count).
  4. Chat: per-invite chat modal preserves messages; real-time via WebSocket/Firebase recommended for production.

3) Turf owner onboarding

Turf owners click "List your turf" → fill the registration form (photos, pricing, amenities, slot rules). They can request a demo. After verification, listing goes live and owner gets a simple dashboard to manage bookings.

Technical architecture & developer notes

Frontend

  • Static HTML/CSS/JS — directly runnable in VS Code. Use modular files: index.html, styles.css, app.js.
  • Light/Dark mode: CSS variables, auto-detect via prefers-color-scheme, toggle to override.
  • Localization: simple JS dictionary for UI strings; expand to JSON per language.
  • Accessibility: use ARIA attributes, alt text for images, and maintain 4.5:1 contrast for readable text.

Realtime & backend (recommended)

  • Session posting / chat / join requests — use a backend API and real-time layer: options:
    • Firebase Realtime DB / Firestore + Auth — fastest to prototype real-time chat and simple auth.
    • Node.js + Socket.IO — good if you want control and a REST API (booking, payments).
  • Bookings: REST endpoints to confirm availability, create booking with idempotency key, then call payment gateway server-side to capture funds.
  • Data model (example): venues, bookings, sessions (game invites), users, messages.

Demo-only (client-only) notes

For demos, store sessions & bookings in localStorage. This is not secure and should only be used for prototypes. Replace with server storage before production.

GitHub Pages & Deployment tips

  • Place index.html at repo root (or set Pages to /docs in settings). If Pages shows a blank page: check console errors, ensure relative paths (no leading /), and ensure resources are present.
  • If your site is an SPA (routes), either use hash-based routing or add a copy of index.html as 404.html to handle refresh navigation.

Security, privacy & moderation

  • Authentication: use secure server-side password hashing (bcrypt) and JWT sessions or provider auth (Google/Apple).
  • Payments: never store card details on your servers. Use PCI-compliant providers (Stripe/Razorpay) and server-side capture.
  • Privacy: minimal PII for quick onboarding. Provide a privacy policy page and give users data deletion options.
  • Moderation: in-app reporting for sessions & messages, and automated content filters for abusive language. Consider rate-limits + CAPTCHA to prevent spam invites.

How Maidaan is different — website experience & offerings

Website Experience (UX & UI differences)

  • Community-first flows: Post & join invites are central (not an afterthought). Many competitors focus mainly on booking and do not make player-matching easy.
  • Transparent pricing: We show final venue fees without platform-added hidden fees. This reduces friction & complaints.
  • Neon-themed, energetic UI: High-contrast neon-dark theme with a light-mode alternative — crafted for a sports vibe that still meets accessibility standards.
  • Integrated playlists: Spotify integration on venue pages — an emotional, experience-driven differentiator that competitors rarely provide.
  • Simple turf onboarding: Turf owners can register quickly with an onboarding flow and request demos — fewer steps than complex vendor portals.

Service & Offering differences

  • Player discovery: Built-in "Find Players" feed and session-based chat — competitors may list venues but lack seamless player matching.
  • Local community features: Team-building, recurring match scheduling, neighborhood leaderboards and event aggregation for local leagues.
  • Owner-friendly policies: Lower commission or subscription models (choose one). Transparent payouts and simple calendar sync.
  • Safety & trust: Verified turf badges, user reviews & ratings, and the ability for turf owners to moderate or confirm players.

Roadmap & priorities (recommended)

  • MVP (0 - 3 months): Booking + simple Turf listing, Find Players posting & join, basic chat via Firebase, payments via Stripe/Razorpay sandbox, English + Hindi UI, light/dark mode, accessibility baseline.
  • Next (3 - 6 months): Turf owner dashboard, repeat bookings, subscription/commission options, better search & map clustering, user verification & reviews.
  • Later (6+ months): League management, tournaments, API for partners, analytics for turf owners, mobile app (React Native / Flutter).

How to test the "Find Players" interactivity (developer -> QA)

  1. Open the site and sign in with demo account (if implemented) or use anonymous posting.
  2. Click + Post Your Game Invite — ensure modal shows fields: sport, date/time, players required, location.
  3. Submit and confirm invite appears immediately in the feed (no refresh).
  4. From another browser or incognito, click "Join" — check that a 'request sent' state is visible and a notification appears for the original poster.
  5. Open "Chat" — send messages; confirm messages are visible to both sides.
  6. Test edge cases: invalid times, duplicate posts, spam protection (rate-limit), and empty fields.
Implementation tip: for a fast demo, use localStorage; for production, use Firestore or Socket.IO + REST API.
